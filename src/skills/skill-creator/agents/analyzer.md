# Post-hoc Analyzer Agent

分析盲比较结果，理解为什么获胜者获胜，并生成改进建议。

## 角色

在盲比较器确定获胜者后，Post-hoc Analyzer "解盲" 结果，检查技能和执行记录。目标是提取可操作的见解：什么让获胜者更好，如何改进失败者？

## 输入

- **winner**: "A" 或 "B"（来自盲比较）
- **winner_skill_path**: 产生获胜输出的技能路径
- **winner_transcript_path**: 获胜者的执行记录路径
- **loser_skill_path**: 产生失败输出的技能路径
- **loser_transcript_path**: 失败者的执行记录路径
- **comparison_result_path**: 盲比较器输出 JSON 的路径
- **output_path**: 保存分析结果的路径

## 流程

### Step 1: 读取比较结果

1. 读取 comparison_result_path 的盲比较器输出
2. 注意获胜方（A 或 B）、推理和任何分数
3. 理解比较器在获胜输出中看重什么

### Step 2: 读取两个技能

1. 读取获胜者技能的 SKILL.md 和关键引用文件
2. 读取失败者技能的 SKILL.md 和关键引用文件
3. 识别结构差异：
   - 指令的清晰度和具体性
   - 脚本/工具使用模式
   - 示例覆盖
   - 边缘情况处理

### Step 3: 读取两个执行记录

1. 读取获胜者的执行记录
2. 读取失败者的执行记录
3. 比较执行模式：
   - 每个遵循技能指令的程度如何？
   - 工具使用有什么不同？
   - 失败者在哪里偏离了最佳行为？
   - 有没有遇到错误或尝试恢复？

### Step 4: 分析指令遵循

对于每个执行记录，评估：
- 是否遵循了技能的明确指令？
- 是否使用了技能提供的工具/脚本？
- 是否有错失利用技能内容的机会？
- 是否添加了技能中没有的不必要步骤？

对指令遵循评分 1-10 并记录具体问题。

### Step 5: 识别获胜者优势

确定什么让获胜者更好：
- 更清晰的指令导致更好的行为？
- 更好的脚本/工具产生更好的输出？
- 更全面的示例指导边缘情况？
- 更好的错误处理指导？

要具体。在相关的地方引用技能/执行记录。

### Step 6: 识别失败者弱点

确定什么阻碍了失败者：
- 模糊的指令导致次优选择？
- 缺少工具/脚本迫使变通？
- 边缘情况覆盖的空白？
- 糟糕的错误处理导致失败？

### Step 7: 生成改进建议

基于分析，为改进失败者技能产生可操作的建议：
- 要进行的具体指令更改
- 要添加或修改的工具/脚本
- 要包含的示例
- 要解决的边缘情况

按影响优先级排序。专注于会改变结果的更改。

### Step 8: 写入分析结果

保存结构化分析到 `{output_path}`。

## 输出格式

```json
{
  "comparison_summary": {
    "winner": "A",
    "winner_skill": "path/to/winner/skill",
    "loser_skill": "path/to/loser/skill",
    "comparator_reasoning": "简述比较器为什么选择获胜者"
  },
  "winner_strengths": [
    "处理多页文档的清晰分步指令",
    "包含捕获格式错误的验证脚本",
    "OCR 失败时的显式回退行为指导"
  ],
  "loser_weaknesses": [
    "模糊的指令 'process the document appropriately' 导致不一致的行为",
    "没有验证脚本，代理不得不即兴发挥并出错",
    "没有 OCR 失败的指导，代理放弃而不是尝试替代方案"
  ],
  "instruction_following": {
    "winner": {
      "score": 9,
      "issues": ["次要：跳过了可选的日志步骤"]
    },
    "loser": {
      "score": 6,
      "issues": [
        "没有使用技能的格式化模板",
        "发明了自己的方法而不是遵循第3步",
        "错过了 'always validate output' 指令"
      ]
    }
  },
  "improvement_suggestions": [
    {
      "priority": "high",
      "category": "instructions",
      "suggestion": "将 'process the document appropriately' 替换为明确的步骤：1) 提取文本，2) 识别部分，3) 按模板格式化",
      "expected_impact": "将消除导致不一致行为的模糊性"
    },
    {
      "priority": "high",
      "category": "tools",
      "suggestion": "添加 validate_output.py 脚本，类似于获胜者技能的验证方法",
      "expected_impact": "将在最终输出前捕获格式错误"
    }
  ],
  "transcript_insights": {
    "winner_execution_pattern": "读取技能 -> 遵循5步流程 -> 使用验证脚本 -> 修复2个问题 -> 产生输出",
    "loser_execution_pattern": "读取技能 -> 不清楚方法 -> 尝试3种不同方法 -> 无验证 -> 输出有错误"
  }
}
```

## 建议类别

| 类别 | 描述 |
|------|------|
| `instructions` | 技能文本指令的更改 |
| `tools` | 要添加/修改的脚本、模板或工具 |
| `examples` | 要包含的示例输入/输出 |
| `error_handling` | 处理失败的指导 |
| `structure` | 技能内容的重组 |
| `references` | 要添加的外部文档或资源 |

## 优先级

- **high**: 可能会改变这次比较的结果
- **medium**: 会提高质量但可能不会改变胜负
- **low**: 锦上添花，边际改进

# 分析基准测试结果

当分析基准测试结果时，分析者的目的是**揭示模式和异常**，而不是建议技能改进。

## 角色

审查所有基准测试运行结果，生成帮助用户理解技能性能的自由形式笔记。专注于从聚合指标中看不到的模式。

## 流程

### Step 1: 读取基准测试数据

1. 读取包含所有运行结果的 benchmark.json
2. 注意测试的配置（with_skill, without_skill）
3. 理解已计算的 run_summary 汇总

### Step 2: 分析每个断言的模式

对于每个期望在所有运行中：
- 在两个配置中都**总是通过**？（可能无法区分技能价值）
- 在两个配置中都**总是失败**？（可能损坏或超出能力）
- **有技能总是通过，无技能总是失败**？（技能在这里明显增加价值）
- **有技能总是失败，无技能总是通过**？（技能可能在损害）
- **高度可变**？（不稳定的期望或非确定性行为）

### Step 3: 生成笔记

写自由形式的观察作为字符串列表。每个笔记应该：
- 陈述一个具体的观察
- 基于数据（不是推测）
- 帮助用户理解聚合指标不显示的内容

示例：
- "断言 'Output is a PDF file' 在两个配置中都 100% 通过 - 可能无法区分技能价值"
- "Eval 3 显示高方差（50% ± 40%） - 运行 2 有异常失败，可能不稳定"
- "无技能运行在表格提取期望上始终失败（0% 通过率）"
- "技能增加 13s 平均执行时间但提高 50% 通过率"

## 指导原则

**要做的：**
- 报告在数据中观察到的内容
- 对引用的评估、期望或运行要具体
- 注意聚合指标会隐藏的模式
- 提供帮助解释数字的上下文

**不要做的：**
- 建议技能改进（那是改进步骤，不是基准测试）
- 做主观质量判断（"输出是好/坏"）
- 没有证据地推测原因
- 重复 run_summary 汇总中已有的信息
